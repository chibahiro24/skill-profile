## 始め方

### 1. リポジトリのフォーク

まず、GitHubでこのリポジトリをフォークします。

### 2. ローカル環境のセットアップ

```bash
# リポジトリのクローン
git clone https://github.com/YOUR_USERNAME/skill-profile.git
cd skill-profile

# 依存関係のインストール
# (プロジェクトの技術スタックに応じて)
npm install  # または yarn install、pip install -r requirements.txt等

# 開発サーバーの起動
npm run dev  # または適切なコマンド
```

詳細なセットアップ手順は[DEPLOYMENT.md](DEPLOYMENT.md)を参照してください。

### 3. ブランチの作成

```bash
# 最新のmainブランチから新しいブランチを作成
git checkout -b feature/your-feature-name
# または
git checkout -b fix/your-bug-fix
```

ブランチ名の規則：
- `feature/` - 新機能の追加
- `fix/` - バグ修正
- `docs/` - ドキュメントの更新
- `refactor/` - リファクタリング
- `test/` - テストの追加・修正
- `plugin/` - プラグインの追加・更新

## コントリビューションの種類

### コードコントリビューション

- **新機能の実装**: 新しい機能やプラグインの追加
- **バグ修正**: 既存の問題の修正
- **パフォーマンス改善**: 最適化やパフォーマンス向上
- **リファクタリング**: コード品質の向上

### ドキュメントコントリビューション

- **ドキュメントの改善**: 誤字脱字の修正、説明の明確化
- **翻訳**: 他言語への翻訳
- **チュートリアル**: 使い方ガイドやサンプルの作成
- **API ドキュメント**: APIリファレンスの整備

### プラグイン開発

- **新しいプラグイン**: AI連携、外部サービス連携プラグインの開発
- **プラグインの改善**: 既存プラグインの機能拡張

詳細は[PLUGIN_DEVELOPMENT.md](PLUGIN_DEVELOPMENT.md)を参照してください。

### その他のコントリビューション

- **Issue報告**: バグレポートや機能提案
- **レビュー**: プルリクエストのレビュー
- **テスト**: テストケースの追加
- **デザイン**: UI/UXの改善提案

## 開発プロセス

### 1. Issue の確認または作成

- 既存のIssueを確認し、重複がないか確認
- 大きな変更の場合は、事前にIssueで議論することを推奨
- Issue テンプレートを使用して報告

### 2. 開発

- コードスタイルガイドに従う
- テストを書く（可能な限り）
- コミットメッセージは明確に

### 3. テスト

```bash
# ユニットテストの実行
npm test

# リンターの実行
npm run lint

# フォーマッターの実行
npm run format
```

### 4. プッシュとプルリクエスト

```bash
# 変更をコミット
git add .
git commit -m "feat: 新機能の説明"

# リモートリポジトリにプッシュ
git push origin feature/your-feature-name
```

その後、GitHubでプルリクエストを作成します。

## コードスタイル

### 一般的なガイドライン

- **可読性**: コードは読みやすく、理解しやすいものにする
- **シンプル**: 必要以上に複雑にしない
- **一貫性**: プロジェクトの既存のスタイルに従う
- **コメント**: 複雑なロジックには説明を加える

### TypeScript / JavaScript

```typescript
// 良い例
export interface SkillProfile {
  userId: string;
  skills: Skill[];
  createdAt: Date;
}

export function calculateSkillLevel(
  usage: number,
  experience: number
): SkillLevel {
  // スキルレベルの計算ロジック
  const score = usage * 0.3 + experience * 0.7;
  return determineLevel(score);
}
```

- ESLint設定に従う
- Prettierでフォーマット
- 型定義を適切に使用
- 関数は小さく、単一責任に
- async/awaitを使用（Promiseチェーンより）

```python
def extract_skills_from_log(log: ActivityLog) -> List[DetectedSkill]:

    """
    skills = []
    # スキル抽出ロジック
    return skills
```

- PEP 8に従う
- Black でフォーマット
- 型ヒントを使用（Python 3.8+）
- Docstringを記述
- テストはpytestを使用

### コミットメッセージ

Conventional Commits形式を使用：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type:**
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響しない変更（空白、フォーマット等）
- `refactor`: バグ修正や機能追加ではないコード変更
- `perf`: パフォーマンス改善
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツールの変更

**例:**

```
feat(plugin): ChatGPTプラグインを追加

- OpenAI APIとの連携機能を実装
- 会話ログからのスキル抽出機能を追加
- ユーザー設定画面を実装

Closes #123
```

## プルリクエストのガイドライン

### プルリクエストを作成する前に

- [ ] コードがビルドできる
- [ ] すべてのテストが通過する
- [ ] リンターエラーがない
- [ ] ドキュメントを更新した（必要な場合）
- [ ] CHANGELOG.mdを更新した（該当する場合）

### プルリクエストのテンプレート

```markdown
## 概要
この変更の目的と内容を簡潔に説明してください。

## 変更内容
- 変更点1
- 変更点2
- 変更点3

## 関連Issue
Closes #123

## テスト方法
この変更をテストする手順を記載してください。

## スクリーンショット（該当する場合）
UI変更がある場合はスクリーンショットを添付してください。

## チェックリスト
- [ ] テストを追加した
- [ ] ドキュメントを更新した
- [ ] コードスタイルガイドに従った
- [ ] 後方互換性を保った（または破壊的変更を明記した）
```

### レビュープロセス

- メンテナーが2営業日以内にレビューします
- 変更が求められた場合は、速やかに対応してください
- レビューコメントには丁寧に対応してください
- 承認されたらメンテナーがマージします

## Issue報告

### バグレポート

Issueテンプレートを使用して以下の情報を含めてください：

- **環境**: OS、ブラウザ、バージョン等
- **再現手順**: バグを再現する詳細な手順
- **期待される動作**: 本来どうあるべきか
- **実際の動作**: 実際に何が起きるか
- **スクリーンショット**: 可能であれば添付
- **エラーメッセージ**: 関連するエラーログ

### 機能提案

- **ユースケース**: なぜこの機能が必要か
- **提案内容**: 具体的にどんな機能か
- **代替案**: 他の解決方法があれば
- **追加情報**: 参考資料やモックアップ等
